## Story

As a Super Admin or Government Official,
I want inferential statistical insights (chi-square tests, correlations, group comparisons, confidence intervals), exportable policy brief PDFs, and a progressive activation system that lights up features as data thresholds are met,
So that I can make evidence-based policy decisions backed by statistical significance tests and share findings with stakeholders who don't use the platform.

## Progressive Activation

This is the **capstone** of the progressive activation pattern. While Story 8.6 introduces the `ThresholdGuard` component, this story builds the full activation infrastructure:

1. **Activation Progress Dashboard**: A dedicated section showing which features are live, which are dormant, and how many submissions remain until each activates
2. **Dormant Hooks**: Phase 5 features (regression, anomaly detection) are registered in the activation system with their thresholds but have no implementation yet — they show as "Coming with 500+ submissions" in the progress dashboard
3. **Self-enabling**: As data accumulates, features transition from dormant -> active without redeployment

## Acceptance Criteria

1. **Given** a Super Admin or Government Official navigates to the "Insights" tab on the Survey Analytics page **When** >= 100 submissions exist **Then** it displays auto-generated inferential results organized in cards:
   - **Association Tests**: 6 chi-square results with chi-square statistic, df, p-value, Cramer's V, plain-English interpretation
   - **Correlations**: 4 correlation results with coefficient, p-value, scatter plot thumbnails
   - **Group Comparisons**: 5 results with test statistic, p-value, group medians box plot
   - **Proportion CIs**: 5 metrics with Wilson score 95% confidence intervals
   - **Enrollment Velocity**: Registration rate forecast with projected date to next threshold
   **And** each card shows a significance badge: green (p < 0.05) or grey (not significant).

2. **Given** the chi-square test for "gender x employment type" **When** computed **Then** the result includes:
   - Chi-square statistic, degrees of freedom, p-value
   - Cramer's V effect size with label (negligible < 0.1, small 0.1-0.3, medium 0.3-0.5, large > 0.5)
   - Plain-English sentence: "There is [a statistically significant / no significant] association between gender and employment type (chi-sq = 45.2, p < 0.001, Cramer's V = 0.23 medium effect)."

3. **Given** a Super Admin or Government Official clicks "Export Policy Brief" **When** the PDF generates **Then** it produces a branded PDF document containing:
   - Header: Oyo State logo, title "Labour Market Intelligence Brief", generation date
   - Executive summary: 3-5 auto-generated key findings
   - Demographics section: key stats in text-based tables
   - Employment section: key stats
   - Skills section: top skills + gap analysis summary
   - Inferential findings: significant associations and correlations
   - Methodology note: sample size, confidence levels, data collection method
   - Footer: "Generated by OSLRS — Oyo State Labour Registration System"
   **And** the PDF is <= 5 pages and downloads immediately (not emailed).

4. **Given** the public `/insights` page **When** >= 200 submissions exist **Then** a "Key Findings" section appears below the existing content displaying 2-3 anonymized inferential results (e.g., "Gender is significantly associated with employment type") with plain-English explanations. No p-values or technical stats shown publicly — just findings.

5. **Given** a Super Admin navigates to the "Equity" tab **When** it loads **Then** it displays 3 additional equity metrics (beyond GPI, youth employment, informal sector from 8.2):
   - Disability employment gap: employed rate for disabled vs non-disabled, with 95% CI
   - Education-employment alignment: % in tier-aligned employment (see mapping in Dev Notes), broken down as aligned / over-qualified / under-qualified
   - Geographic equity (Gini coefficient): distribution of registrations across LGAs, with interpretation (0 = perfect equality, 1 = total concentration)

6. **Given** any analytics page (all roles) **When** the user scrolls to the bottom **Then** an "Analytics Activation Status" collapsible panel shows:
   - A list of all analytics features grouped by status: Active (green), Approaching (amber, > 50% of threshold), Dormant (grey)
   - For each feature: name, current N, required N, progress bar
   - Phase 5 features show: "Requires 500+ submissions — advanced regression models will activate automatically"

7. **Given** the backend `GET /analytics/insights` endpoint **When** called **Then** it computes all inferential tests in a single request, caches the result (Redis, TTL 1 hour), and returns all test results with per-section thresholds.

8. **Given** fewer than 100 submissions exist **When** a user opens the Insights tab **Then** each section shows its own threshold guard with progress toward activation (sections with lower thresholds may still display while others are dormant).

9. **Given** the backend `GET /analytics/activation-status` endpoint **When** called by any dashboard role **Then** it returns the current submission count and the full feature activation registry with met/unmet status — no expensive inferential computation, just count checks.

### Statistical Library Choice

Using `simple-statistics` (~15KB gzipped) per spec recommendation (section 3.5 Option B). It provides:
- `sampleCorrelation` (Pearson's r)
- `linearRegression` / `linearRegressionLine` (for enrollment forecast)
- `chiSquaredDistributionTable` (critical value lookup — NOT a CDF function)
- Basic descriptive stats (`mean`, `median`, `standardDeviation`)

**Functions to implement from formulas** (straightforward rank-based or formula computations):
- Chi-square independence test + p-value (see formula below)
- Cramer's V effect size
- Spearman correlation (rank-transform + Pearson)
- Mann-Whitney U (rank-sum + normal approximation)
- Kruskal-Wallis H (rank-sum + chi-square approximation)
- Wilson score CI
- Gini coefficient (sorted proportions method)

### P-Value Computation Strategy

`simple-statistics.chiSquaredDistributionTable` is a **lookup table** of critical values at specific alpha levels (0.10, 0.05, 0.025, 0.01, 0.005) for df 1-100. It does NOT compute arbitrary p-values.

**Recommended approach**: Use the table for **significance bracket determination**:
- If chiSq > table[df][0.005] -> `p < 0.001` (beyond table range, effectively)
- If chiSq > table[df][0.01] -> `p < 0.01`
- If chiSq > table[df][0.05] -> `p < 0.05`
- Else -> `p >= 0.05`

This is sufficient for plain-English interpretations ("statistically significant at the 0.01 level"). The `pBracket` field carries this string. The `pValue` field is set to the midpoint of the bracket or -1 if only bracket is available. The interpretation generator uses `pBracket` (not the raw `pValue`) for text output.

For exact p-values (enhancement): implement the regularized upper incomplete gamma function (`1 - gammainc(df/2, chiSq/2)`) using the series expansion. This is ~20 lines of code.

### Chi-Square Independence Test Implementation

```ts
function chiSquareIndependence(observed: number[][]): { chiSq: number; df: number } {
  const rowSums = observed.map(row => row.reduce((a, b) => a + b, 0));
  const colSums = observed[0].map((_, j) => observed.reduce((sum, row) => sum + row[j], 0));
  const total = rowSums.reduce((a, b) => a + b, 0);

  let chiSq = 0;
  for (let i = 0; i < observed.length; i++) {
    for (let j = 0; j < observed[0].length; j++) {
      const expected = (rowSums[i] * colSums[j]) / total;
      if (expected > 0) {
        chiSq += Math.pow(observed[i][j] - expected, 2) / expected;
      }
    }
  }
  return { chiSq, df: (observed.length - 1) * (observed[0].length - 1) };
}
```

### Wilson Score Interval

```ts
function wilsonCI(successes: number, n: number, z = 1.96): { lower: number; upper: number } {
  const p = successes / n;
  const denominator = 1 + z * z / n;
  const centre = p + z * z / (2 * n);
  const margin = z * Math.sqrt((p * (1 - p) + z * z / (4 * n)) / n);
  return {
    lower: (centre - margin) / denominator,
    upper: (centre + margin) / denominator,
  };
}
```

### Education-Employment Alignment Mapping

The survey form has `education_level` (select_one: none, primary, secondary, vocational, tertiary, postgraduate) and `employment_type` (select_one: wage_public, wage_private, self_employed, family_unpaid, apprentice, contractor). There is no `job_sector` field.

**Concrete operationalization** — tier-based comparison:

| Field | Tier 1 (Basic) | Tier 2 (Intermediate) | Tier 3 (Advanced) |
|-------|---------------|----------------------|-------------------|
| Education | none, primary | secondary, vocational | tertiary, postgraduate |
| Employment | family_unpaid, apprentice | self_employed | wage_private, wage_public, contractor |

**Classification**:
- **Aligned**: education_tier == employment_tier (education level matches employment formality)
- **Over-qualified**: education_tier > employment_tier (e.g., tertiary degree holder doing unpaid family work)
- **Under-qualified**: education_tier < employment_tier (e.g., primary education in formal wage employment)

**Only applied to employed respondents** (unemployed have no employment tier to compare).

This is a pragmatic proxy metric — not a direct skill-to-job match. The story should display it with a subtitle: "Proxy based on education level vs employment formality tier" to manage interpretation expectations.

### Public Key Findings Architecture

The key findings flow is **decoupled** to protect public endpoint performance:

```
[SA/Official loads /analytics/insights]
  -> StatisticalTestsService computes all tests (~3-5s)
  -> Results cached: analytics:insights:{scope} (TTL 1hr)
  -> Key findings extracted (top 2-3 significant, plain-English)
  -> Written to: analytics:public:key-findings (TTL 1hr)

[Visitor loads /public/insights]
  -> PublicInsightsService reads analytics:public:key-findings from Redis
  -> If exists AND totalSubmissions >= 200: include keyFindings[]
  -> If missing: keyFindings = undefined (section not shown)
  -> Public endpoint remains ~500ms (no inferential computation)
```

**Trade-off**: Key findings only appear on the public page AFTER an SA/Official has loaded the Insights tab at least once (populating the cache). This is acceptable because:
1. SA/Officials typically access the dashboard daily
2. The 1hr TTL means findings refresh with each SA/Official visit
3. Showing no findings is better than spiking the public endpoint to 5s

### Performance Budget

Inferential computation is the most expensive analytics operation:
- Single SQL query extracts ALL needed columns (~200ms for 10K rows)
- 6 chi-square tests + 4 correlations + 5 group comparisons + 5 CIs (~500ms in-memory)
- Total: ~1-3s for 10,000 submissions

**Mitigation**:
- Single extraction query (not 15+ separate queries)
- Statistical computations happen in-memory on the extracted dataset
- Result cached for 1 hour
- Activation status endpoint is separate and lightweight (~50ms)

### Anti-Patterns to Avoid

1. **Do NOT compute statistics in SQL** — SQL is for data extraction only. Statistical tests run in Node.js
2. **Do NOT use `jstat` or `mathjs`** — too heavy. `simple-statistics` + hand-rolled formulas per spec section 3.5
3. **Do NOT render chart images in PDF** — PDFKit can't easily render Recharts components server-side. Use text tables and formatted numbers
4. **Do NOT expose p-values publicly** — public page gets plain-English findings only
5. **Do NOT implement Phase 5 features** — only register them as dormant hooks in the activation system
6. **Do NOT run inferential tests below minimum N** — return threshold guard response, never misleading results on small samples
7. **Do NOT import StatisticalTestsService into PublicInsightsService** — use Redis cache bridge (see Public Key Findings Architecture)
8. **Do NOT assume `chiSquaredDistributionTable` is a CDF** — it's a static lookup table of critical values, not a continuous distribution function
9. **Do NOT modify existing `EquityMetrics.tsx`** — create `ExtendedEquityMetrics.tsx` alongside it
