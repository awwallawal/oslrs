# Story 1.5.5: Support Section (5 Pages)

Status: done

## Story

As a **Public Visitor or Registered User**,
I want to **find answers to common questions and contact support**,
so that I can **resolve issues without confusion**.

## Acceptance Criteria

1. **AC1: Support Landing Page (/support)**
   - GIVEN a visitor navigates to `/support`
   - WHEN the page renders
   - THEN the page MUST display:
     - H1: "How Can We Help?"
     - Subheading explaining available support options
     - Search box (Phase 1: UI only, placeholder - no functionality)
     - **Quick Links** section with 4 cards:
       - FAQ card with icon → `/support/faq`
       - Guides card with icon → `/support/guides`
       - Contact Us card with icon → `/support/contact`
       - Verify Worker card (full-width) with icon → `/support/verify-worker`
     - **Popular Questions** section showing top 5 FAQs as expandable items:
       1. How do I register for OSLSR?
       2. What is the NIN and where do I get one?
       3. Is registration free?
       4. How long does verification take?
       5. Can I update my information after registering?
     - "See all FAQs →" link
   - AND quick link cards match the FeatureCard/BenefitCard pattern

2. **AC2: FAQ Page (/support/faq)**
   - GIVEN a visitor navigates to `/support/faq`
   - WHEN the page renders
   - THEN the page MUST display:
     - H1: "Frequently Asked Questions"
     - Search box (Phase 1: UI only placeholder)
     - Category filter tabs: [All] [Registration] [Survey] [Verification] [Marketplace] [Privacy] [Technical]
     - FAQ sections by category with accordion items:
       - **Registration** (5 questions): How to register, Is it free, NIN info, Photo requirement, Unemployed eligibility
       - **Survey** (3 questions): Duration, Save progress, What info requested
       - **Verification** (3 questions): Duration, Badge meaning, ID card question
       - **Marketplace** (3 questions): What is it, Opt-in requirement, Visible info
       - **Privacy & Data** (3 questions): NIN storage, Data deletion, Who sees data
       - **Technical Issues** (3 questions): Email not received, Password reset, Survey not loading
     - "Can't find your answer?" CTA → `/support/contact`
   - AND reuses FAQAccordion component from Participate feature

3. **AC3: Guides Page (/support/guides)**
   - GIVEN a visitor navigates to `/support/guides`
   - WHEN the page renders
   - THEN the page MUST display:
     - H1: "Guides"
     - Subheading: "Step-by-step instructions to help you use OSLSR"
     - **For Workers** section with 4 guide cards:
       - How to Register
       - How to Complete the Survey
       - How to Opt Into the Marketplace
       - How to Get a NIN
     - **For Employers** section with 3 guide cards:
       - How to Search the Marketplace
       - How to Create an Employer Account
       - How to Verify a Worker
     - "Need More Help?" CTA linking to FAQ and Contact
   - NOTE: Phase 1 guide cards link to placeholders or external resources (detailed guide content is Phase 2)

4. **AC4: Contact Page (/support/contact)**
   - GIVEN a visitor navigates to `/support/contact`
   - WHEN the page renders
   - THEN the page MUST display:
     - H1: "Contact Us"
     - Subheading: "Get in Touch with the OSLSR Team"
     - **General Inquiries** section:
       - Email: support@oslsr.oyo.gov.ng
       - Phone: +234 XXX XXX XXXX (placeholder)
       - Hours: Monday - Friday, 8:00 AM - 5:00 PM
       - Response time: "We typically respond within 2-3 business days"
     - **Technical Support** section:
       - Email: tech@oslsr.oyo.gov.ng
       - What to include in reports
     - **Data & Privacy Requests** section:
       - Email: dpo@oyostate.gov.ng
       - Address: Ministry of Labour & Productivity, State Secretariat, Ibadan
       - NDPA 30-day response note
     - **Partnerships & Media** section:
       - Email: partnerships@oyostate.gov.ng
     - **Report Abuse** callout box (warning variant):
       - Email: report@oslsr.oyo.gov.ng
       - Scam warning: "OSLSR registration is FREE"
     - **Office Location** section with address (map placeholder for Phase 2)

5. **AC5: Verify Worker Page (/support/verify-worker)**
   - GIVEN a visitor navigates to `/support/verify-worker`
   - WHEN the page renders
   - THEN the page MUST display:
     - H1: "Verify a Worker"
     - Subheading explaining the verification tool purpose
     - **Verification Lookup** section:
       - Input field for verification code (format: OSLSR-XXXX-XXXX)
       - "Verify →" button
       - On submit: redirect to `/verify-staff/{code}` (existing verification page)
     - **What Does Verification Mean?** callout box:
       - What it confirms (3 points): registered, NIN validated, identity confirmed
       - What it does NOT confirm (3 points): skill level, work quality, employment history
       - "Always interview and assess workers" disclaimer
     - "Need Help?" section with FAQ and Contact links
   - NOTE: The actual verification API call is handled by existing `/verify-staff/:id` page (Story 1-6)

6. **AC6: Routing & Navigation Integration**
   - GIVEN the current App.tsx has `/support` as a single PlaceholderPage
   - WHEN implementing the support section
   - THEN MUST:
     - Convert `/support` to nested route structure
     - Add routes: `/support` (index), `/support/faq`, `/support/guides`, `/support/contact`, `/support/verify-worker`
     - All routes lazy-loaded with Suspense
   - AND navigation links remain correct:
     - NavDropdown.tsx Support link → `/support` (no change needed)
     - MobileNav.tsx Support link → `/support` (no change needed)
     - Footer.tsx Support and Contact Us links → `/support` (no change needed)

7. **AC7: Responsive Design & Accessibility**
   - GIVEN various viewport sizes
   - WHEN viewing any Support section page
   - THEN all pages MUST:
     - Be fully responsive (mobile-first)
     - Meet WCAG 2.1 AA accessibility standards
     - Use proper heading hierarchy (only ONE H1 per page)
     - Have sufficient color contrast (4.5:1 minimum)
     - Support keyboard navigation
     - FAQ accordions and category tabs must be keyboard accessible

8. **AC8: Testing Requirements**
   - GIVEN the feature test folder pattern
   - WHEN implementing tests
   - THEN tests MUST:
     - Be co-located in `apps/web/src/features/support/__tests__/`
     - Cover all 5 pages with render and content tests
     - Verify navigation links work correctly
     - Test FAQ accordion and category filter interaction
     - Test verify-worker form submission/redirect
     - Achieve minimum 80% coverage for the feature
   - ESTIMATED: ~30-40 tests total (5 pages × 6-8 assertions each)

### Category Tabs Pattern

Use shadcn Tabs component if installed, otherwise create simple tab pattern:

```tsx
// Check if installed
// npx shadcn@latest add tabs

// Or create simple CategoryTabs:
interface CategoryTabsProps {
  categories: string[];
  activeCategory: string;
  onCategoryChange: (category: string) => void;
}

function CategoryTabs({ categories, activeCategory, onCategoryChange }: CategoryTabsProps) {
  return (
    <div className="flex flex-wrap gap-2 mb-8">
      {categories.map((category) => (
        <button
          key={category}
          onClick={() => onCategoryChange(category)}
          className={cn(
            'px-4 py-2 rounded-full text-sm font-medium transition-colors',
            activeCategory === category
              ? 'bg-primary-600 text-white'
              : 'bg-neutral-100 text-neutral-700 hover:bg-neutral-200'
          )}
        >
          {category}
        </button>
      ))}
    </div>
  );
}
```
