## Story
As the **launch operator driving public traffic**, I want the **public self-serve wizard to render a SHORT "Public Core" form** while **enumerators/clerks keep the FULL baseline instrument**, so that **public fatigue drops and completion rises тАФ without fragmenting the registry**: every channel still aggregates into one `respondents` table and every dashboard reads it honestly.

## Context & Why
The pinned public form is the **full ~42-question baseline instrument** (a labour-force STUDY fused with a skills REGISTRY). That's the fatigue driving the 292 abandoned drafts (esp. Step-4 Labour). But the two collection purposes have different epistemics: the **baseline study** needs the full instrument on a **representative enumerator sample**; the **public self-serve** is a **convenience sample** that should be a fast REGISTRY sign-up. Fix = **split by channel**, not delete data. The mechanism already exists (public reads the pin; enumerators pick from published) тАФ this story enables + verifies it, and keeps aggregation honest via the shared-key contract. Draft form: `docs/launch-campaign/oslsr-public-core-v1.xlsx` (~16 adult-visible Qs; reuses the master's exact names/choice-lists; **includes `email`**, which the current pinned form lacks).

### Dedup verification (2026-07-04) тАФ the wizard-native vs form-question overlap
The public wizard collects 11 identity/contact/consent fields natively (Steps 1тАУ2) and hides matching Step-4 questions via `WIZARD_PROVIDED_FIELD_NAMES` + `computePrefill` (`apps/web/src/features/registration/pages/Step4Questionnaire.tsx`). Cross-checked every Public Core field against that map:
- **Deduped correctly (hidden тАФ asked once, in the wizard):** `surname`(familyName), `firstname`(givenName), `gender`, `dob`, `phone_number`(phone), `email`, `nin`, `consent_marketplace`, `consent_enriched`. Choice fields map cleanly: `gender_list=[male,female,other]` (wizard `prefer_not_to_sayтЖТother`), consents `yes_no` (boolean `trueтЖТyes`). тЬУ
- **NOT a duplicate (form-only; wizard doesn't collect them):** `consent_basic`, `main_occupation`, `employment_type`, `years_experience`, `skills_possessed`, `skills_other`, guardian block. тЬУ
- **DUPLICATE тЖТ fix (AC7):** `lga_id`. Wizard stores `lgas.id` (UUID); form `lga_list` uses slugs тЖТ `mapWizardValueToChoice` fails тЖТ shown twice. **Fix = remove `lga_id` from the Public Core form.**

**Root-cause fix тЖТ Story 13-16 (`13-16-lga-value-canonicalization`).** The LGA UUIDтЖФslug mismatch is deeper than the dedup: `respondents.lgaId` is `text` (no FK) and holds the **UUID** on the public/wizard path but the **slug** on the enumerator path (verified vs prod: 139 UUID public + 1 slug enumerator), so the analytics `l.code = r.lga_id` join FAILS for all 139 public rows (they render as UUID/"Unknown"). 13-16 canonicalizes to the **slug** everywhere (wizard writes `lga.code`, backfills the 139). **This story (13-14) stays independent:** removing `lga_id` from the Public Core is the immediate pre-launch, form-only fix; 13-16 is the systemic cure (keep both тАФ belt-and-suspenders; do NOT make 13-14 depend on 13-16). Enumerators are unaffected on the DUPLICATE (they fill the FULL form directly, no wizard) but ARE part of the value-split 13-16 fixes.

**2026-07-04 тАФ 13-16 IMPLEMENTED (code+backfill script; prod backfill = operator residual). The `lga_id` removal here is now defense-in-depth, not the sole fix.** тЪая╕П **NEW CONSTRAINT from the 13-16 AC5 audit тАФ `lga_list` choice VALUES:** the live master form's `lga_list` diverges from `lgas.code` for **6/33 LGAs** (`ibadan_ne/nw/se/sw`, `ogbomoso_north/south` vs `ibadan_north_east/тАж`, `ogbomosho_*` тАФ fossil vocabulary from `scripts/generate-xlsform.cjs`). Any form that KEEPS an `lga_id` question (the full/Baseline master; the Public Core's retained-but-unused `lga_list`) writes a non-joining value for those 6 LGAs on the enumerator path. **Before publishing/pinning either form (Task 2), align the 6 `lga_list` choice values to `lgas.code`** тАФ labels stay human, only the slug values change. (Old submissions holding the alias values in `raw_data` are unaffected тАФ analytics read `respondents.lga_id`.)
