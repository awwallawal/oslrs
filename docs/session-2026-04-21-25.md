**Decision:** Nullable NIN with partial unique index + status enum (`active | pending_nin_capture | nin_unavailable | imported_unverified`) instead of (a) blanket FR21 removal or (b) separate `external_beneficiaries` table.

**Why:**

## Operational lessons (memorise these)

### Lesson 1: sshd_config drop-in override "first value wins"

`/etc/ssh/sshd_config.d/*.conf` are loaded in alphanumeric order. SSH config uses **first-match-wins** semantics. A directive in `50-cloud-init.conf` overrides the same directive in `60-cloudimg-settings.conf` AND in the main `sshd_config`. **Always grep all drop-ins after editing sshd_config.**

```bash
sudo grep -rE "^(PermitRootLogin|PasswordAuthentication|PubkeyAuthentication)" /etc/ssh/sshd_config.d/
```

### Lesson 2: Ubuntu SSH service name is `ssh`, not `sshd`

Debian/Ubuntu uses `ssh.service`. Red Hat / CentOS / Fedora use `sshd.service`. Always:

```bash
sudo systemctl reload ssh         # Ubuntu
sudo systemctl reload sshd        # RHEL family
```

### Lesson 3: DO Console is SSH-based, not hypervisor-OOB

DO Web Console works by establishing an SSH session from DO's own infrastructure IP ranges (e.g. `162.243.0.0/16`) to your droplet's port 22, then bridging to noVNC. **If you firewall SSH to operator-only sources, Console breaks** — and the failure mode looks like browser/ISP filtering, not firewall blocking.

DO Recovery Console likely shares this dependency (verification pending).

True hypervisor-OOB break-glass: **DO Snapshot restore** (always works regardless of firewall).

### Lesson 4: PM2 startup hook is required for survive-reboot

By default PM2 does not auto-start on boot. Run once:

```bash
pm2 startup    # prints a sudo command — copy and run it (registers pm2-root.service via systemd)
pm2 save       # snapshots current process list as resurrection set
```

After reboot, `systemd` brings up `pm2-root.service` which runs `pm2 resurrect` from the saved dump.

### Lesson 5: Docker containers need explicit restart policy

Default `docker run` policy is `no` (don't restart). For production:

```bash
docker update --restart=unless-stopped <container>...
```

`unless-stopped` survives reboots but respects manual `docker stop`. Alternative `always` survives even manual stops.

### Lesson 6: GitHub Actions deploys cannot live inside a tailnet-only firewall

`appleboy/ssh-action` runs from GitHub-hosted runner public IPs. Tailnet-only firewall blocks all CI deploys. Either:
- Widen firewall (Option A) — accept defence-in-depth degradation
- Move CI to self-hosted runner inside tailnet (Option B) — more work, cleaner

### Lesson 7: Postal-gov reports — "write as if delivered, hold until true"

Awwal's stated pattern for Ministry-facing deliverables: write the report assuming completion, gate the actual submission on a Deliverable Commitment Matrix verifying every claim. Don't ship aspirational claims; don't ship half-done either.

### Lesson 8: BMAD correct-course workflow is the discipline restorer

When scope decisions accumulate in conversation without flowing through canonical artifacts, run `correct-course` (incremental mode). Outputs a Sprint Change Proposal that cleanly routes to PM → Architect → UX → SM in sequence. Resists the "absorb scope inline" temptation that breaks downstream traceability.

### Terminology decision (2026-04-26)

## Postscript 2 — Phase 2 dual-domain ship (2026-04-26 evening, commit `9ece951`)

Right after the super_admin fossil purge, executed the next item on the pending-follow-up list: making `https://oyoskills.com` serve the full app alongside `https://oyotradeministry.com.ng`.

### Strategic decision — Strategy A (relative API URLs)

Two paths considered: (A) bake `VITE_API_URL` to a relative path so XHR is always same-origin, or (B) keep absolute URL + add multi-origin CORS plumbing. Picked A — eliminates an entire class of dual-domain CORS headaches because browsers don't enforce CORS on same-origin requests. Multi-origin parsing was still needed for socket.io and CSP `connect-src` (the WS layer can't piggy-back on same-origin handling), so `CORS_ORIGIN` became comma-separated, but the HTTP API path is now CORS-irrelevant.

### Mechanics

- **DNS** — Cloudflare Console: A record `oyoskills.com → 159.89.146.93`. `www` was already a CNAME to `oyoskills.com` from Phase 1 email setup, so it inherits — no second A record needed (and Cloudflare prevents two record types at the same name anyway).
- **Cert** — `sudo certbot certonly --nginx --cert-name oyotradeministry.com.ng -d oyotradeministry.com.ng -d www.oyotradeministry.com.ng -d oyoskills.com -d www.oyoskills.com --expand --non-interactive`. SAN list 2 → 4. Path unchanged at `/etc/letsencrypt/live/oyotradeministry.com.ng/`. Auto-renew config inherits new SAN; new expiry 2026-07-25.
- **Repo changes (8 files)**:
  - `.github/workflows/ci-cd.yml` — `VITE_API_URL=/api/v1` (hardcoded; removed `${{ vars.VITE_API_URL }}` lookup so a stale GH Variable cannot override)
  - `apps/api/src/app.ts` — `corsOrigins` exported array (comma-split + per-entry validation: no wildcard, must be `http(s)://`); `wsUrls` array; `cors({ origin: corsOrigins })`; `cspDirectives.connectSrc` spreads `wsUrls`
  - `apps/api/src/realtime/index.ts` — socket.io `cors.origin` uses parsed list
  - `apps/api/src/__tests__/csp-parity.test.ts` — `PROD_WS_URL` → `PROD_WS_URLS` (array); dev-fallback `ws://localhost:3000` expands to full prod list via `flatMap`
  - `apps/web/src/hooks/useRealtimeConnection.ts` — `getSocketUrl()` returns `window.location.origin` when `VITE_API_URL` is relative or unset
  - `infra/nginx/oslsr.conf` — both server blocks `server_name` cover all 4 domains; HTTP redirect uses `$host` (preserves requested domain) instead of `$server_name` (would collapse to first listed name); CSP `connect-src` adds `wss://oyoskills.com`
- **VPS .env (PRE-deploy, per SEC-3 lesson)**: `PUBLIC_APP_URL=https://oyoskills.com`, `CORS_ORIGIN=https://oyotradeministry.com.ng,https://oyoskills.com`. Pitfall #6 leading-whitespace artifact on `PUBLIC_APP_URL=` line caught and stripped via `[[:space:]]*` regex anchor.

### Verification

- **All 4 domains** return `HTTP 200` on `/`. Both `/api/v1/health` return clean JSON.
- **Cert SAN confirmed** via `openssl s_client -connect oyoskills.com:443` showing all 4 DNS entries.
- **Browser proof of Strategy A**: DevTools Network tab on a request from `https://www.oyoskills.com/` to `/api/v1/auth/refresh` showed `Sec-Fetch-Site: same-origin` — meaning the browser never even considered this a cross-origin call. CORS is dormant by construction.
- **CSP delivered correctly**: `connect-src 'self' wss://oyotradeministry.com.ng wss://oyoskills.com ...` in live response headers.
- **Zero CSP violations** from our policy after hard-refresh. Pre-deploy violations in PM2 logs (calls from `www.oyoskills.com` to `https://oyotradeministry.com.ng/api/v1/...`) were captured during the brief window where the cert was valid for oyoskills.com but the new code/nginx weren't yet deployed — purged by hard-refresh dropping the old absolute-URL bundle.
- **PM2 healthy** post-deploy: online, 59.5 MB, 0% CPU. ↺ counter incremented by 1 (graceful sigint → restart). No crash-loop on the new comma-separated CORS_ORIGIN parser.

### Operational lessons

1. **CSP violations during cert-only-but-no-config-yet windows are noise, not bugs.** Cert was expanded ~25 min before the code/nginx deploy. In that window, browsers visiting `https://oyoskills.com` got a TLS-valid response from the default server (oyotradeministry's block) and ran the OLD JS bundle, which blew up against the OLD CSP. After deploy + hard refresh, zero violations. Future dual-domain rollouts: minimise the cert-only window or expect this transient noise.
2. **Strategy A is dramatically simpler than multi-origin CORS** when the HTTP API is the only same-origin-eligible path. Reserve multi-origin CORS for cases where you genuinely need the API hosted on a separate origin from the web app.
3. **`$host` vs `$server_name` in nginx multi-domain redirects** — the latter collapses to the first listed `server_name` token, breaking the redirect target on secondary domains. Always use `$host` for "preserve what the client asked for".
4. **GH Actions Variables can silently override workflow defaults**. Removing the `${{ vars.X || 'fallback' }}` lookup and hardcoding the literal value is the safer move when "fallback" is actually the intended behaviour for all environments.

After Phase 2 dual-domain shipped, surfaced that Cloudflare was already proxying `oyoskills.com` (`Server: cloudflare` + `CF-RAY` headers in responses) — the proxy toggle was flipped at some point during the day, ahead of the planned Phase 3 step. Verified the rest of Phase 3:

- **SSL/TLS mode = Full (strict)** — CF→VPS connection uses HTTPS with cert validation; our VPS cert (oyotradeministry.com.ng SAN list expanded to 4 domains in Phase 2) covers oyoskills.com so strict mode works
- **Always Use HTTPS** + **Automatic HTTPS Rewrites** = ON (CF edge issues 301 for plain HTTP requests; the VPS-side nginx `return 301 https://$host$request_uri` is now redundant for oyoskills.com but stays as belt-and-suspenders for direct-to-VPS oyotradeministry traffic)
- **WAF Managed Rules** = Always Active (Cloudflare Free tier baseline ruleset)

- **WebSocket through CF proxy** verified — Browser DevTools Network tab on `wss://www.oyoskills.com/socket.io/?EIO=4&transport=websocket` returns status 101 Switching Protocols; `Server: cloudflare` + our `X-Proxy-Upstream: socketio` header both present, confirming chain client → CF → nginx → socket.io intact

### Real-client-IP plumbing — two-step fix

**Discovery:** post-Cloudflare, `req.ip` was resolving to CF edge IPs (e.g., `172.69.22.77` in `172.64.0.0/13`) rather than the real client. Effects: rate limiting buckets all CF-edge-served users together; audit logs show CF edge IPs; IP-based blocking would block CF edges.

**Step 1 (commit `4c2d909`)** — Updated `app.set('trust proxy', ...)` from `1` to `['loopback', 'linklocal', 'uniquelocal', ...CLOUDFLARE_IPS]`. Created `apps/api/src/constants/cloudflare-ips.ts` with 15 IPv4 + 7 IPv6 CIDRs from `cloudflare.com/ips` (re-verify annually). Theory: Express's X-Forwarded-For walking would skip past CF edge IPs and stop at the real client.

**Result: did not work.** Post-deploy verification showed captcha.verified + auth.login_success still logging CF edge IPs. Root cause: Cloudflare's X-F-F population is inconsistent — sometimes the real client appears in the chain, sometimes only the CF edge. When only CF edge is present, Express's walking finds nothing untrusted to stop at and returns the last trusted IP (the CF edge).

**Step 2 (commits `bf98931` + `1383373`)** — Created `apps/api/src/middleware/real-ip.ts` that reads `CF-Connecting-IP` directly. Cloudflare sets this header on every proxied request with the real client IP and overwrites any client-supplied value. Security guard: only honor the header when the immediate sender (X-Real-IP set by nginx) is in CLOUDFLARE_IPS — prevents spoofing by attackers reaching the VPS directly bypassing Cloudflare. Falls through to Express's trust-proxy + X-F-F default for non-CF traffic (oyotradeministry.com.ng direct).

`bf98931` failed CI's strict `tsc` build (vitest's tsx transpile-only didn't catch it locally) — `proxy-addr` ships without bundled types and the compiled trust function expects `(addr, i)` not `(addr)`. `1383373` added `@types/proxy-addr@^2.0.3` and the second arg.

**Verification (2026-04-27 ~07:00 UTC):** captcha.verified + auth.login_success events on pid 39949 log real client IP `197.211.52.65` matching the laptop's public IP from `icanhazip.com`. Trust-proxy whitelist from step 1 stays in place as defense-in-depth for direct-to-VPS oyotradeministry traffic.

### Operational lessons (additional)

5. **`pnpm build` is required pre-commit when touching `.ts` files.** vitest uses tsx (transpile-only) so type errors silently slip past local test runs. CI's `tsc` is strict and catches them. Always run `pnpm --filter @oslsr/api build && pnpm --filter @oslsr/web build` before commit on TypeScript changes. Future improvement: wire `tsc --noEmit` into pre-commit hook.
6. **`CF-Connecting-IP` is the canonical real-client-IP header behind Cloudflare**, not X-Forwarded-For. CF documents this — use it directly. The trust-proxy + X-F-F dance is a fallback for non-CF deployments.
7. **Cloudflare proxy auto-injects content into `robots.txt`.** When proxied (orange-cloud), CF prepends a "Cloudflare Managed Content" block to robots.txt with their bot-blocking directives (Amazonbot, ClaudeBot, GPTBot, etc.). Visible only in HTTP responses, not in the source file. This is automatic behavior and not a security concern, but it's a "what the heck added this?" surprise on first encounter.
8. **DO NOT proxy email DNS records (MX/SPF/DKIM/DMARC).** Proxying MX records breaks SMTP delivery instantly. Proxying TXT (SPF/DKIM/DMARC) breaks Resend's domain verification chain. These must stay DNS-only/grey-cloud forever — even if Phase 3 takes the website to orange-cloud. The 3-vendor email split (ImprovMX inbound + Resend outbound) is independent of Cloudflare's proxy state.
